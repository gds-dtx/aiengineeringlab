# Code review for AI-assisted development

How to review code generated by GitHub Copilot, Claude, Gemini Code Assist, and Amazon Kiro effectively, including what to look for and what each tool commonly gets wrong.

## Who this applies to

This guide applies to Quality Assurance (QA) engineers reviewing pull requests, QA leads setting review standards, and champions coaching teams on effective review practices.

## The reviewer's mindset

### Treat AI code like code from a new team member

Code generated by GitHub Copilot, Claude, Gemini Code Assist, or Amazon Kiro should be treated the same way you would treat code from a talented but unfamiliar contributor. It may follow good patterns, but it does not know your codebase, your team's conventions, your security requirements, or your production constraints.

The reviewer's job is not just to check whether the code works. It is to check whether it belongs in your codebase.

### You must understand every line

Do not commit code you cannot explain line by line. All four tools can produce plausible-looking code that uses patterns you have never seen. If you cannot explain what a block of code does and why, do not approve it. Ask the author to explain it, or investigate it yourself.

Pay particular attention to error handling logic, as all four tools often add catch blocks that silently swallow exceptions. Security-related code needs close scrutiny, including authentication, authorisation, input validation, and data sanitisation. Check database queries carefully. AI may produce queries that work on small datasets but perform badly at scale. Do the same for concurrency patterns, where AI-generated async code frequently introduces race conditions.

## What AI gets right and why you still need to check

GitHub Copilot, Claude, Gemini Code Assist, and Amazon Kiro are generally good at producing syntactically valid code. They follow common patterns for the language and framework, include reasonable structure and naming, and handle the happy path well.

This is why review is critical. The code looks correct, passes a casual scan, and may even pass basic tests. The problems live in the gaps between what the AI was asked for and what production actually requires.

## Common categories of issues

### Shallow error handling

All four tools tend to generate error handling that catches exceptions without doing anything meaningful. Be aware of empty catch blocks and generic error messages that provide no useful diagnostic information. Watch for catch-all handlers that mask specific failure modes.

Does the error handling give the operations team enough information to diagnose the problem. Error handling that swallows context makes monitoring and incident response significantly harder.

### Missing edge cases

AI tools typically handle obvious cases well but miss boundary conditions. Common gaps include:

- empty collections
- null or undefined inputs
- concurrent access scenarios
- maximum field lengths
- Unicode and special character handling
- timezone edge cases

Think about the inputs users would never send intentionally but that your system will inevitably receive.

### Security blind spots

None of the four tools know your threat model. They frequently generate code with:

- insufficient input validation
- Structured Query Language (SQL) or NoSQL injection vulnerabilities
- insecure defaults for authentication or session management
- missing rate limiting
- overly permissive Cross-Origin Resource Sharing (CORS) configurations

Run all AI-generated code through your security scanning toolchain regardless of how secure it looks on inspection. Treat it as you would any third-party code.

### Performance assumptions

All four tools generate code that works at demo scale but may struggle in production. Common issues include:

- n+1 query patterns
- unbounded collection processing
- missing pagination
- unnecessary object creation in loops
- synchronous operations that should be asynchronous

Consider what happens when this code processes 10,000 records instead of 10.

### Inconsistency with codebase patterns

AI tools follow general language conventions rather than your team's specific patterns. This results in:

- different naming conventions
- unfamiliar architectural patterns
- alternative library usage for functionality you already have
- inconsistent error response formats

Check that the solution aligns with your team's agreed approaches. Confirm it reuses existing patterns rather than introducing new ones without justification.

### Accessibility gaps in generated user interface code

When any of the four tools generate front-end code, they frequently omit or incorrectly implement accessibility requirements. Common issues include:

- missing Accessible Rich Internet Applications (ARIA) attributes
- incorrect heading hierarchies
- insufficient colour contrast
- absent keyboard navigation support

Check that AI-generated User Interface (UI) code includes semantic HTML, keyboard operability, and screen reader support. Colour contrast should be 4.5 to 1 or above for normal text to meet Web Content Accessibility Guidelines (WCAG) 2.2 Level AA.

### Test quality issues

All four tools can generate tests with high coverage but low value. Be aware of:

- tests that merely assert the code runs without checking behaviour
- overly specific assertions tied to implementation details rather than outcomes
- missing negative test cases
- mocked dependencies that hide integration problems

If you changed the implementation, would these tests catch a real regression?

### Open source readiness

AI-generated code may include:

- hardcoded configuration values
- placeholder secrets or API keys
- comments containing internal system details
- references to proprietary libraries

Before committing, check that the code is suitable for an open repository and that anything sensitive has been moved to environment variables or secrets management.

## Review workflow for AI-assisted pull requests

### Step 1: Context check

Before reading the code, understand which tool generated it and what prompt or context was provided. If the pull request description does not explain this, ask the author. The quality of output from all four tools depends heavily on the quality of the prompt and context.

### Step 2: Architecture review

Review whether the change fits the existing architecture. All four tools frequently introduce new patterns or libraries when existing ones would suffice. Check that the solution aligns with your team's agreed approaches.

### Step 3: Line-by-line with focus areas

Review the code with particular attention to the categories listed above. Spend extra time on error handling paths, security-relevant code, database interactions, and external service calls.

### Step 4: Run the tests critically

Do not just check that tests pass. You should read the test code, make sure the assertions are meaningful, ensure that the tests cover failure scenarios and would catch a regression if the implementation changed.

### Step 5: Check automated scan results

Verify that the code has passed through your Continuous Integration and Continuous Deployment (CI/CD) quality gates. These include linting, static analysis, security scanning, and test coverage thresholds. If your pipeline does not include these, see the [quality gates and CI/CD](quality-gates-ci-cd.md) guide.

### Step 6: Accessibility review for user interface changes

For any changes that affect the user interface, verify that the code meets WCAG 2.2 Level AA requirements. Use automated tools such as axe and Lighthouse as a starting point. Manual testing with keyboard navigation and screen readers is also necessary.

## Tool-specific review considerations

### GitHub Copilot
- validate that Copilot-generated regex patterns are not vulnerable to Regular expression Denial of Service (ReDoS) attacks
- check that Copilot-suggested authentication flows handle token expiry and session fixation correctly
- review Copilot-generated SQL queries for parameterisation across all database interactions
- verify that Copilot-generated test assertions check behaviour, not just that code runs

### Claude
- use Claude to reason through edge cases and threat models that automated tools cannot assess
- ask Claude to explain any generated code blocks that are unclear before approving them
- Claude is particularly effective at reviewing complex multi-step workflows and personal data handling logic
- use Claude to produce plain English summaries of security findings for wider team review

### Gemini Code Assist
- validate that Gemini-generated Application Programming Interface (API) integrations handle authentication errors and token expiry correctly
- check that Gemini-generated error responses do not expose stack traces or internal system details
- review Gemini-generated configuration files for permissive default settings
- verify that Gemini-generated UI components meet accessibility requirements

### Amazon Kiro
- review Kiro-generated agentic workflows carefully, as multi-step automation can introduce logic errors that are harder to trace
- validate that Kiro-generated dependency suggestions do not introduce known Common Vulnerabilities and Exposures (CVEs) or incompatible licences
- check that Kiro-generated CI/CD pipeline steps do not inadvertently expose secrets or credentials
- verify that Kiro-generated repeatable tasks behave consistently across environments

## Coaching your team

When providing feedback on AI-generated code, focus on the why rather than just the what. Engineers are learning to work with GitHub Copilot, Claude, Gemini Code Assist, and Amazon Kiro. Understanding why a particular suggestion is problematic helps them write better prompts. It helps them make better judgements in the future.

For example, instead of "This error handling is wrong", you could say "Kiro has generated a generic catch block here. In our codebase, we need to handle database connection failures separately from validation errors because our monitoring system categorises alerts differently. Here is how we typically structure this..."

Document your team's specific review expectations for AI-assisted code. Include examples of code you have accepted and rejected from each tool, with explanations. Add these to your team's contributing guide and reference them in pull request templates. Your review standards should evolve as your team's experience with all four tools matures.
